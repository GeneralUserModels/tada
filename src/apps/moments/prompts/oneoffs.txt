## Updating existing tasks

When your analysis surfaces a task that substantially overlaps with one of the existing tada tasks above, do NOT write a new candidate to `{logs_dir}/oneoffs/`. Instead, edit the existing task file directly using `write_file` at:

  `{logs_dir}/../logs-tada/<topic>/<slug>.md`

You may change anything in that file — title, description, body, frequency, schedule, confidence, usefulness, trigger — but **keep the slug/filename the same** (slug is the task's identity across run history, results, and user state). Edits to existing tada files bypass the filter stage and re-execute on the next scheduler tick. Do not also write a candidate for the same idea — pick one path or the other.

Rule of thumb for "substantially overlaps": same user intent + overlapping data sources + similar resulting artifact. Slight differences in framing are NOT enough to justify a new task; refine the existing one.

If your candidate is genuinely distinct from every existing task, write it as a new candidate to `{logs_dir}/oneoffs/` per the normal flow.

## Trigger-based one-offs

Some one-offs are only useful once a particular condition becomes true (a new email from a sender appears, today is the day before a flight, a file matching a pattern arrives in a watched folder, a calendar event happens). For these, include an OPTIONAL `trigger:` field in the frontmatter with a single-sentence natural-language condition.

Triggered one-offs are evaluated daily by a separate trigger-check stage; they re-fire each day the condition is currently true. Do **not** write a `trigger:` for one-offs that should fire as soon as discovered — leave the field off and the one-off will be filtered + executed normally.

Examples of good triggers:
- `trigger: a new email from stripe arrives in the user's inbox`
- `trigger: today is within 48 hours of a calendar event whose location is outside the user's home city`
- `trigger: a new pdf paper appears in ~/Downloads with "transformer" in the filename`

Bad triggers (too vague to evaluate from logs):
- `trigger: when the user is stressed`
- `trigger: when the project is going badly`

The AI agent that will execute these tasks is powerful. It can:
- Search the web, crawl pages, and fetch live data
- Browse authenticated websites (GitHub, Gmail, Twitter/X, YouTube, Slack, Google Docs, etc.) using the user's Chrome cookies
- Do deep research and analysis — read papers, compare approaches, synthesize across dozens of sources
- Read files on the user's machine for context
- Run shell commands, scripts, git operations, Python/Node code
- Pre-draft emails, Slack messages, documents, and code for the user to review
- Generate reports, build slide decks, create static HTML interfaces to present results
- Spawn subagents to parallelize work across different data sources

The agent CANNOT: call LLMs at runtime in its output, modify arbitrary files on the user's machine, or build interactive interfaces that require a backend. It produces static artifacts (HTML, markdown, drafts) that the user reviews.

**Prioritize tasks that amplify the user's abilities:**
- **Information foraging and synthesis** — the agent's biggest advantage is reading, comparing, and synthesizing across many sources faster than any human. Prioritize tasks where the agent does deep research the user wouldn't have time for.
- **Complex multi-step work** — prefer ambitious tasks that chain together many operations. The agent thrives on tasks that are too tedious or time-consuming for a human to bother with.
- **Preparation and context-building** — upcoming meetings, ongoing projects, open questions — the agent can do the legwork so the user walks in prepared.

## Time awareness

Consider the current date/time and what's coming up for the user (calendar, emails, etc.) — timely tasks are often more useful than generic ones.

## What to look for

Focus on what the user is CURRENTLY doing or interested in. Look across ALL sources — session logs, emails, calendar, notifications, filesystem. Look for:
- **Active research** — the user is searching for, reading about, or comparing something specific. The agent can do deeper research and compile a structured summary.
- **Planning or decision-making** — the user is planning a trip, evaluating options, comparing products, etc. The agent can gather more information, build a comparison, or draft an itinerary.
- **Unfinished work** — something the user started but hasn't completed. The agent can pick it up and finish it or prepare a draft.
- **Draft responses** — emails, Slack messages, or PR reviews the user needs to reply to. The agent can prepare draft responses the user can review and send in seconds instead of writing from scratch.
- **Information the user needs** — based on upcoming calendar events, recent emails, or notifications, what context or preparation would be helpful?
- **Content the user is consuming** — videos watched, articles read, threads followed. The agent can summarize, extract key points, or find related resources.
- **Learning opportunities** — things the user is working with but could understand more deeply. The agent can teach concepts, explain techniques, surface relevant papers/talks, or create personalized explainers based on what the user is actually doing right now.

## Quality bar

- Every task MUST be grounded in specific activity observed in the logs — not generic advice.
- Each task should produce a concrete artifact (summary, analysis, draft, comparison, report).
- Tasks should be specific enough that an agent can execute them without further clarification.
- Do NOT produce fluffy or vague tasks. "Help user be more productive" is garbage. "Research the top 5 restaurants near [location from calendar event]" is good.
- Do NOT produce tasks that are really recurring automations — those belong in discover.py.

1. **Inventory what's already implemented** (see section above): read every file in `{logs_dir}/oneoffs/`, `{logs_dir}/tasks/`, and `{logs_dir}/../logs-tada/` plus the `_moment_state.json` signals. Build the "already-covered" list.

3. **Read & analyze**: Use subagents to read ALL session_*/labels.jsonl files in parallel. Also read email, calendar, notifications, and filesys logs.
4. **Identify opportunities**: Based on what you find, identify specific one-off tasks where the agent can help right now — only ones not already covered.
5. **Write tasks**: Write each task file using write_file. Before each file, explicitly check the already-covered list and confirm the task is genuinely new.
