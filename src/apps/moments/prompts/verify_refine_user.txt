## Files

- `{output_dir}/index.html` — entry point, loads CSS and JS
- `{output_dir}/app.js` — main application logic with DATA object at top
- `{output_dir}/styles.css` — app-specific styles
- `{output_dir}/data.js` — data file (may not exist if data is in app.js)

- `{output_dir}/meta.json` — metadata

## What counts as an error (fix these)

1. **Syntax errors** — run `node --check` on every `.js` file. Fix anything that fails to parse.
2. **Runtime-breaking references** — undefined variables, wrong property paths, JS reading `item.title` when DATA uses `item.name`, missing files referenced from `index.html`.
3. **Shared runtime misuse** — `components.js` exposes `window.PN`; app code may safely declare local helpers like `const h = React.createElement` and `const {{ useState }} = React`. Only fix actual runtime collisions or missing references.
4. **Broken loads** — `index.html` must correctly reference all required JS and CSS files in the right order.

After any fix, re-run `node --check` on the changed files to confirm they parse.

## What NOT to do

- Do NOT improve layout, spacing, design, or styling.
- Do NOT add edge-case handling, empty-state polish, or new features.
- Do NOT rewrite or restructure working code.
- Do NOT modify `components.js` or `base.css`.
- Do NOT delete `index.html`. Even if the app looks low-quality, as long as it runs without errors, leave it alone.

## Rules

- Use `edit_file` for surgical fixes, not full file rewrites.
- Preserve the existing design, data, and content exactly. Your only job is to make sure the app runs without errors.
