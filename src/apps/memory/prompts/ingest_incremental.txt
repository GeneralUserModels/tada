
## Incremental Ingest

This is a RE-RUN. The last ingest was on **{last_ingest_date}**.

### New inputs since last ingest:

{new_inputs_list}

Read the new inputs across all source types — first-person sources (active conversations, chats with assistant, audio transcripts, Tada moment feedback) carry the highest signal, but every source can move the wiki forward. Then do ALL of the following without restraint. The common failure mode of incremental runs is touching only a handful of obviously-affected pages; push past that.

**1. Create new pages for any genuinely new entity or theme.** A new person named, a new project surfaced, a new place / idea / tool / concept that doesn't exist in the wiki yet — give it its own page as soon as you have a real paragraph of content. Do not wait for "enough evidence" to accumulate across runs; the page can start lean and grow. The wiki should be growing in breadth, not just depth. The "no stub pages" rule from the main prompt only blocks truly speculative entries with no real content — not real-but-thin ones.

**2. Update existing pages broadly — not only the ones the new data names.** New data that directly touches an existing page → obviously update it. But also:
- Re-read pages **adjacent** to the new data. A new conversation about a project pulls through to that project's collaborator pages, related-work pages, the user's overview page, the work-patterns page, etc. Trace the wiki-links outward.
- Look for **stale framings** the new data implicitly contradicts or refines, even on pages you would not normally revisit this run.
- Use web search / browser to **enrich** existing pages independently of the new logs — verify facts on people pages, pull in fresh context on projects, follow links the user has been visiting. Some edits each run should be enrichment edits with no direct new-log trigger.
- Adjust confidence scores wherever new data reinforces or contradicts existing claims.

**3. Revisit older data with fresh eyes.** The "new since last ingest" listing above is your starting point, NOT your boundary. Earlier runs almost certainly missed things — entities never written up, themes never crystallized, connections never drawn. Each incremental run, sample older data the wiki has under-covered:
- Walk the {logs_dir}/ tree (`ls`, `find`) and look for **directories or sources the wiki has nothing about yet** — a folder with no corresponding page, a stream of audio files that's never been mined, an old chat session that was never read. Read into them.
- Pick a few **older sessions / conversations / audio files** at random that predate the last ingest and skim them for entities, places, or patterns missing from the wiki.
- Cross-check the wiki against itself: scan `index.md` for thinly-covered areas (folders with one page, pages with very low confidence, pages last updated long ago) and go pull more evidence from the historical logs to deepen them or to spawn the sub-pages they imply.
- New pages discovered from older data are just as welcome as new pages from new data — note in the log entry that they came from a backfill pass.

If a page has lost topical coherence — e.g. it's become a catch-all for loosely related things — split it into focused sub-pages before adding more. See "Splitting overgrown pages" in the main prompt.
